03
核心亮点 Key Innovations
01

OpenClaw 多 Agent 协作架构

4 个独立 Agent（qa-bot / wiki-manager / weekly-reporter / card-builder）通过 OpenClaw 框架实现职责分离与能力组合。单一职责 Prompt 更聚焦，独立部署可单独升级，故障隔离，权限物理隔离。

单一职责 · 独立演进 · 故障隔离
02

符号化知识编译（零向量库依赖）

不采用传统 RAG 向量检索，而是让 LLM 在 ingest 阶段把知识编译成结构化 Markdown 页面（实体/概念/来源），带 [[Wikilink]] 交叉引用。查询时直接读取完整页面做综合。

纯 Markdown · 零外部依赖 · 可解释
03

自动化更新数据底座

wiki-manager 维护 5 步同步流水线：wiki-discover → wiki-sync → base-sync → wiki-ingest → wiki-graph。content_hash 增量判断，未变更文件跳过，原始镜像与编译知识物理隔离。

增量同步 · 物理隔离 · 定时调度
04

可交互前端卡片

知识图谱子图可拖拽、可缩放、可点击查看详情（vis.js）。周报卡片一屏读完，部门看板一眼掌握全局。weekly-reporter 调用 card-builder 将 Markdown 渲染为飞书消息卡片直接推送到群聊。

交互式图谱 · 结构化卡片
05

Scope-Aware 权限隔离设计

根据聊天群自动推断 scope，用户甚至不用登录。更精细的是数据分级：公开信息（PM 姓名等）可跨项目回答；敏感 KPI（GMV、转化率等）严格按 scope 拒答。

拒答不是 Prompt 层面的"请求 LLM 不要说"，而是物理层面的"工具层面就读不到数据"——data_query.py 在读取前会检查 scope，跨项目数据直接不加载。

群即边界 · 数据分级 · 物理隔离

🔎 增量拉取逻辑

• 比对本地 sync_state.json（doc_token + edit_time）
• 新增文档：首次下载 + 记录元数据
• 修改文档：edit_time 变化则覆盖更新
• 删除检测：远端缺失 → 标记 DELETED
• 生成增量报告：新增/修改/删除计数

🔬 检索机制

Step 0: 权限过滤 → scope 过滤可见目录
Step 1: 关键词匹配 wiki/index.md（CJK 双字滑动窗口）
Step 2: 图邻居扩展（graph.json，confidence ≥ 0.7）
Step 3: 读取相关页面完整内容
Step 4: LLM 综合答案（标注 [[PageName]]）
Step 5: 返回带飞书链接的 Markdown

risk-monitor 检测到风险 → 嵌入周报 → PM 周会讨论 → 产生行动项 → 下周检查完成状态 → 风险解除或升级
