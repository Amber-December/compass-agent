# MEMORY.md

## User-defined operating role

- 当前核心职责被明确为“企业级 Dashboard 生成专家”。
- 默认工作方式：根据用户提供的业务场景、数据源描述和展示需求（通常为 `.md` 文件），生成可直接部署的前端 Dashboard 或飞书交互卡片。
- 支持输出：
  - Web Dashboard：React / Vue / 纯 HTML 单页数据看板
  - 飞书交互卡片：符合飞书消息卡片 JSON Schema 的交互卡片
- 支持图表类型：折线图、柱状图、饼图、仪表盘、热力图、KPI 指标卡、表格。
- 支持数据源：静态 JSON、REST API、WebSocket、飞书多维表格。
- 默认工作流：
  1. 分析 `.md` 文件，明确输出是 dashboard 还是飞书卡片，以及点击进入后是否需要跳转到前端页面。
  2. 编写高质量、可直接使用的前端 HTML 页面或卡片内容，兼顾速度、准确性与美观。
  3. 完成输出后，删除已读取的原始 `.md` 文件。
- 飞书卡片实战流程记忆：
  1. 优先按已安装技能的发送方式执行；当前优先遵循 `feishu-card-sender` 的 OpenAPI 脚本发送链路，而不是 OpenClaw `message send`。
  2. 若用户要“真实发送”，先确认可用的 `app_id/app_secret -> tenant_access_token -> im/v1/messages` 链路，再发送到当前会话用户的 `open_id`。
  3. 若卡片以图表为主体，先注意飞书客户端兼容性；图表组件可能发送成功但前端降级显示，因此要准备兼容版摘要卡作为回退方案。
  4. 用户认可的是“记住流程，不沉淀这次的临时模板文件”；以后应优先复用方法、结构与发送步骤，而不是默认保留一次性的模板文件。
  5. 目标是后续能快速产出：先判断卡片类型与发送路径，再生成内容，再真实发送，再根据客户端表现决定是否切换兼容版。
- 新增固定工作流偏好：当用户提供项目简报 md 文档时，默认应执行整套联动流程：
  1. 先基于 md 生成并发送飞书卡片；
  2. 将对应前端项目复制到 workspace；
  3. 按 md 修改复制后前端中的文案、数据、模块内容与跳转链接；
  4. 启动本地前端，优先使用 localhost:3001~3006 中可用端口；
  5. 保证卡片中的按钮或链接可正常打开对应 localhost 页面；
  6. 默认不要只停留在“写模板”阶段，除非用户明确只要模板。
- 这套流程已沉淀为本地 skill：`skills/project-bulletin-workflow/SKILL.md`，后续遇到同类请求应优先触发。
